# Decision 1: Shell Detection in ~/.tsuku/env

## Question

How does `$TSUKU_HOME/env` detect which shell it's running in to source the right init cache file?

## Context

`$TSUKU_HOME/env` is a static shell script sourced from `.bashrc` and `.zshenv` by the tsuku installer. It currently only exports PATH via:

```sh
export PATH="${TSUKU_HOME:-$HOME/.tsuku}/bin:${TSUKU_HOME:-$HOME/.tsuku}/tools/current:$PATH"
```

The fix requires it to also source the appropriate per-shell init cache file. The cache system produces two separate files:
- `$TSUKU_HOME/share/shell.d/.init-cache.bash` — for bash
- `$TSUKU_HOME/share/shell.d/.init-cache.zsh` — for zsh

The cache files are generated by `internal/shellenv/cache.go` (`RebuildShellCache`). Each cache file contains shell init snippets collected from installed tools. The Go code in `cmd/tsuku/shellenv.go` already solves this for the `eval $(tsuku shellenv)` path — it reads `$SHELL` and picks the right cache file. But `$TSUKU_HOME/env` is a static shell script that must work without spawning a subprocess.

Key constraints:
- No subprocess overhead (no `$(tsuku shellenv)` or command substitution)
- Must work in both bash and zsh (same file is sourced by both)
- Must fail safely when the cache doesn't exist (CI/Docker environments with no installed tools)
- Must be POSIX-enough to not break exotic setups
- The solution becomes `envFileContent` in `internal/config/config.go` and must stay in sync with `website/install.sh`

The installer adds `. "$TSUKU_HOME/env"` to `.bashrc` (bash) and `.zshenv` (zsh), meaning the env file is sourced by whichever shell the user is running — bash or zsh — and must detect which one at the time of sourcing.

## Options Evaluated

### Option A: `$BASH_VERSION` / `$ZSH_VERSION` variables

Check the shell-set variables `$BASH_VERSION` and `$ZSH_VERSION`. Both bash and zsh set their respective version variable automatically when a shell process starts. Neither variable is set in the other shell.

```sh
if [ -n "$BASH_VERSION" ]; then
  _tsuku_cache="${TSUKU_HOME:-$HOME/.tsuku}/share/shell.d/.init-cache.bash"
elif [ -n "$ZSH_VERSION" ]; then
  _tsuku_cache="${TSUKU_HOME:-$HOME/.tsuku}/share/shell.d/.init-cache.zsh"
fi
[ -n "$_tsuku_cache" ] && [ -f "$_tsuku_cache" ] && . "$_tsuku_cache"
unset _tsuku_cache
```

**Pros:**
- Zero subprocess overhead — pure variable reads
- Reliable: both variables are set unconditionally by their respective shells, including all non-login and non-interactive modes
- POSIX-compatible syntax (`[ -n "$VAR" ]`); works in bash, zsh, and any POSIX sh
- Handles the "neither bash nor zsh" case cleanly by leaving `$_tsuku_cache` unset and skipping sourcing
- File-existence guard (`[ -f ... ]`) naturally handles CI/Docker environments
- Idiomatic — used by many shell frameworks (e.g., oh-my-zsh uses this pattern)

**Cons:**
- A user who manually sources this file from a different shell (e.g., `sh`, `dash`) gets no init cache, which is correct behavior (no cache file for those shells exists)
- A user who deliberately sets `$BASH_VERSION` or `$ZSH_VERSION` as regular environment variables could mislead detection — extremely unlikely and not a security concern here

**Verdict:** Chosen — most reliable, zero overhead, standard pattern, clean fallback.

### Option B: `$0` / process name parsing

Parse `$0` or `/proc/$$/exe` to determine the shell.

```sh
_tsuku_shell=$(basename "$0")
case "$_tsuku_shell" in
  bash|-bash) _tsuku_ext="bash" ;;
  zsh|-zsh)   _tsuku_ext="zsh" ;;
esac
```

**Pros:**
- Doesn't rely on shell-defined variables

**Cons:**
- `$0` is unreliable when a file is sourced: in bash, `$0` is the name of the script that sourced the file, not `bash`. When `.bashrc` sources `$TSUKU_HOME/env`, `$0` reflects the sourcing script's name or the shell invocation name — behavior varies across bash and zsh and across interactive vs. non-interactive modes
- `/proc/$$/exe` is Linux-only; breaks on macOS
- Requires `basename` subprocess (violates "no subprocess overhead" constraint)
- More fragile: `$0` is sometimes set to `-bash` (login shell), `bash`, or even the calling script's path — all require special-casing

**Verdict:** Rejected — `$0` is unreliable when sourcing, and `/proc/$$/exe` is not portable. Violates the no-subprocess and reliability constraints.

### Option C: Source both, rely on shell-specific guards in cache files

Write a single entry point that sources both cache files unconditionally. Each cache file would need internal guards (e.g., `[ -n "$BASH_VERSION" ]`) to avoid running the wrong content.

**Pros:**
- Simple source call in env file (no detection logic)

**Cons:**
- The actual cache files are generated by `internal/shellenv/cache.go`. Their content is collected from tool-provided shell init scripts (via `install_shell_init`). Requiring every tool's init script to guard itself is an invasive constraint that's hard to enforce and breaks the separation of concerns
- Sourcing bash syntax in zsh (or vice versa) will cause parse errors — the cache wraps content in `( ... ) 2>/dev/null || true` to isolate errors, but a syntax error in zsh from bash-specific syntax (e.g., `local -n` nameref, `declare -A` associative arrays) can still cause visible warnings or misbehavior depending on zsh's error suppression behavior
- Fundamentally unsafe: bash and zsh have meaningful syntax differences that cannot always be hidden behind error suppression

**Verdict:** Rejected — requires trust that all tool init scripts are cross-shell compatible, which is not guaranteed. The cache architecture is explicitly per-shell for good reason.

### Option D: Single unified cache file in POSIX-compatible syntax

Instead of per-shell caches (`.init-cache.bash`, `.init-cache.zsh`), always write a single `.init-cache.sh` that all shells can source.

**Pros:**
- No detection needed in `$TSUKU_HOME/env` — always source one file
- Simpler cache rebuild logic

**Cons:**
- `internal/shellenv/cache.go` generates per-shell caches today, and `install_shell_init` writes per-shell files (e.g., `tool.bash`, `tool.zsh`). Changing to a single cache would require tools to only provide POSIX-compatible init scripts — ruling out bash-specific completions, zsh-specific `compdef` calls, and other shell-specific features that many popular tools use
- This is a significant architectural change with broad impact on the recipe ecosystem
- The current per-shell design exists precisely to allow shell-specific initialization

**Verdict:** Rejected — changes the cache architecture and restricts tools to POSIX-only init syntax. The per-shell design is intentional.

### Option E (emerged from code reading): Reuse `$SHELL` variable as in `shellenv.go`

The Go code in `cmd/tsuku/shellenv.go` uses `$SHELL` to detect the shell. Apply the same logic in shell script.

```sh
_tsuku_shell=$(basename "${SHELL:-}")
case "$_tsuku_shell" in
  bash) _tsuku_ext="bash" ;;
  zsh)  _tsuku_ext="zsh" ;;
esac
[ -n "$_tsuku_ext" ] && [ -f "${TSUKU_HOME:-$HOME/.tsuku}/share/shell.d/.init-cache.$_tsuku_ext" ] && \
  . "${TSUKU_HOME:-$HOME/.tsuku}/share/shell.d/.init-cache.$_tsuku_ext"
```

**Pros:**
- Consistent with the Go implementation in `shellenv.go`

**Cons:**
- `basename "$SHELL"` spawns a subprocess (`basename` is an external command) — violates the no-subprocess overhead constraint
- `$SHELL` reflects the user's configured login shell, not necessarily the shell currently executing the script. If a bash user runs a zsh script that sources env, or vice versa, `$SHELL` would return the wrong answer
- `$BASH_VERSION`/`$ZSH_VERSION` are more accurate because they reflect the actually-running interpreter, not the user's preference setting

**Verdict:** Rejected — spawns a subprocess and reads the wrong variable for in-process detection. `$BASH_VERSION`/`$ZSH_VERSION` are strictly more accurate.

## Chosen Approach

**Option A: `$BASH_VERSION` / `$ZSH_VERSION` shell-set variables**

The updated `envFileContent` constant becomes:

```sh
# tsuku shell configuration
# Add tsuku directories to PATH
export PATH="${TSUKU_HOME:-$HOME/.tsuku}/bin:${TSUKU_HOME:-$HOME/.tsuku}/tools/current:$PATH"

# Source shell init cache if available
if [ -n "$BASH_VERSION" ]; then
  _tsuku_init_cache="${TSUKU_HOME:-$HOME/.tsuku}/share/shell.d/.init-cache.bash"
elif [ -n "$ZSH_VERSION" ]; then
  _tsuku_init_cache="${TSUKU_HOME:-$HOME/.tsuku}/share/shell.d/.init-cache.zsh"
fi
# shellcheck disable=SC2154
[ -n "${_tsuku_init_cache:-}" ] && [ -f "$_tsuku_init_cache" ] && . "$_tsuku_init_cache"
unset _tsuku_init_cache
```

The `unset` at the end avoids polluting the shell environment with the temporary variable.

## Key Implementation Details

- **Variables checked**: `$BASH_VERSION` (set by bash) and `$ZSH_VERSION` (set by zsh) — both are always set when the respective shell is running, regardless of interactivity or login mode
- **Temporary variable name**: `_tsuku_init_cache` — underscore prefix signals internal use, avoids collisions
- **File existence guard**: `[ -f "$_tsuku_init_cache" ]` before sourcing — safe in CI/Docker where no tools are installed yet and no cache exists
- **Cache file paths**:
  - bash: `$TSUKU_HOME/share/shell.d/.init-cache.bash`
  - zsh: `$TSUKU_HOME/share/shell.d/.init-cache.zsh`
- **Sync requirement**: This content must be kept in sync between:
  - `internal/config/config.go` (`envFileContent` constant)
  - `website/install.sh` (the `cat > "$ENV_FILE"` heredoc, lines 120-124)
- **Other shells**: If the env file is sourced from fish, dash, or sh, neither `$BASH_VERSION` nor `$ZSH_VERSION` will be set, `_tsuku_init_cache` stays empty, and sourcing is skipped — correct behavior since no cache file for those shells exists

## Assumptions

- `$BASH_VERSION` is never set as a user-defined environment variable in zsh sessions (and vice versa for `$ZSH_VERSION`). This is a reasonable assumption: these are shell-reserved variables not present in typical environment configs.
- The cache file format generated by `RebuildShellCache` remains shell-specific (one file per shell) — the existing architecture is preserved.
- The env file must remain valid POSIX sh for sourcing from contexts where `set -euo pipefail` may be active (the file itself doesn't use pipefail, but callers might). The `[ -n ... ] && [ -f ... ] && .` pattern is POSIX-safe.
- Fish shell users who manually source this file get no init cache, which is acceptable since fish uses a different mechanism and isn't sourced from `.bashrc`/`.zshenv` by the installer.

## Confidence

**High.** `$BASH_VERSION` and `$ZSH_VERSION` are well-established, documented shell builtins. The pattern is widely used in cross-shell scripts (oh-my-zsh, nvm, etc.). All constraints are satisfied: no subprocess, works in both bash and zsh, fails safely when cache absent, POSIX-compatible syntax. The rejected options each have concrete, objective flaws against the stated constraints.
