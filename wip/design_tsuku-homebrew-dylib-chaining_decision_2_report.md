# Decision 2: Recipe interface for declaring chained dylib dependencies

## Question

How do recipes declare what dylibs to chain into the binary's load path?

- **(a)** Reuse `metadata.runtime_dependencies`
- **(b)** Reuse per-step `dependencies`
- **(c)** Introduce a new explicit field (e.g., `chained_lib_dependencies` at metadata or step level)

## Context recap

Recipe schema today (`internal/recipe/types.go`):

- `MetadataSection.Dependencies` (line 169) â€” recipe-wide install-time deps; replaces step-derived install set in `actions/resolver.go:179`.
- `MetadataSection.RuntimeDependencies` (line 170) â€” recipe-wide runtime deps; replaces step-derived runtime set in `resolver.go:187`.
- `MetadataSection.ExtraDependencies` / `ExtraRuntimeDependencies` â€” extend variants.
- `Step.Dependencies` (line 389) â€” step-scoped install-time deps; replaces action implicit when set (`resolver.go:117`).

How `RuntimeDependencies` is consumed today:

1. `actions/resolver.go` walks it into `ResolvedDeps.Runtime` (a separate map from `InstallTime`).
2. `executor/plan_generator.go:704` only iterates `deps.InstallTime` to expand dependency plans. `Runtime` is **not** expanded into install plans.
3. `install/manager.go:53,137-149` consumes `RuntimeDependencies` as `map[string]string` to gate "wrapper script vs symlink" â€” purely for the launch shim. It never reaches `ExecutionContext`.
4. `actions/homebrew_relocate.go:609-615` reads `ctx.Dependencies.InstallTime` (NOT `Runtime`) when building dylib RPATHs. Gated on `Type == "library"` (line 92, 103).

So the "wiring bug" the exploration calls out is real: even if you write `runtime_dependencies = [...]` and lift the `Type == "library"` gate, the chaining walk reads `InstallTime`, which `RuntimeDependencies` never populates.

The wget recipe duplicates because the step-level `dependencies = [...]` (line 36) is the only thing that actually triggers install + flows into `ctx.Dependencies.InstallTime`. The metadata-level `runtime_dependencies = [...]` (line 7) is what `install/manager` reads to decide it needs wrappers. They are doing different jobs through different paths â€” the duplication is a symptom, not the design.

The recipes survey:

- 323 recipes use `runtime_dependencies`. Most are auto-generated by the homebrew builder (`builders/homebrew.go:2049,2145,2238`) and exist purely to drive wrapper-script generation. Almost none of those want dylib chaining.
- `wget.toml` is the unique manual case combining metadata `runtime_dependencies` with step-level `dependencies` to force install-time presence of the chained libs.
- `curl.toml` uses metadata-level `dependencies` (install-time, recipe-wide) and pairs it with explicit `set_rpath`. Explicit comment defers the macOS path to "until runtime_dependencies supports dylib chaining."
- `tmux.toml` punts darwin entirely with a comment naming the same gap.

## Option (a): Reuse `metadata.runtime_dependencies`

### TOML shape

`wget.toml` rewritten:

```toml
[metadata]
name = "wget"
runtime_dependencies = ["openssl", "gettext", "libidn2", "libunistring"]

[[steps]]
action = "homebrew"
formula = "wget"
when = { os = ["linux"], libc = ["glibc"] }

# ... no separate step-level dependencies needed
```

`tmux.toml` (new darwin path):

```toml
[metadata]
name = "tmux"
runtime_dependencies = ["libevent", "utf8proc"]

[[steps]]
action = "homebrew"
formula = "tmux"
```

### Flow to ExecutionContext

Requires three changes:

1. `executor/plan_generator.go:704` â€” also expand `deps.Runtime` into `DependencyPlan`s so the chained libs actually get installed (currently only `InstallTime` is expanded).
2. `executor/executor.go:964-973` â€” populate `ctx.Dependencies` with **both** `InstallTime` and `Runtime` (or merge runtime into install for the chaining walk).
3. `actions/homebrew_relocate.go:103,609` â€” lift the `Type == "library"` gate and read whichever map ends up carrying chained deps.

### Backward compatibility

**Risky.** 323 recipes already declare `runtime_dependencies`. For most of them (npm wrapper recipes, etc.) the meaning was "install these and wrap binaries to put them on PATH." Repurposing the same field to also drive macOS dylib RPATH walking would:

- Force install of every package wrappers reference even when no chaining is needed. Mostly fine on Linux (already happens via wrappers) but introduces install-time cost.
- For the `homebrew_relocate` action, it would attempt to add `@rpath` entries to bottle dylibs against directories that may not contain matching dylibs (e.g., a `nodejs` dependency for a wrapper-based recipe). Worst case: harmless extra rpaths; best case: still wasted work. Need to verify no recipe currently relies on `runtime_dependencies` resolving to a non-tsuku-installable target.
- The hidden semantic shift ("runtime_dependencies now means dylib chain too") is invisible to current authors.

### Author ergonomics

- One field instead of two â€” strict win over today's wget pattern.
- "Runtime dependencies" is the closest natural-language match for "libs my binary loads at runtime." Authors who think "I need libssl.3.dylib at run time" already reach for this name.
- BUT the field is overloaded. Today it also drives PATH-shim wrappers for npm/pip-style tools. Authors writing a homebrew recipe and a wrapper recipe both write `runtime_dependencies = [...]` but get different behavior. That overload is already there; (a) just adds a third meaning.

### Validator implications

- Verify each entry is a known recipe (already done).
- New: warn when `runtime_dependencies` contains a recipe whose `Type != "library"` â€” chained RPATHs against a tool's `lib/` are usually meaningless. Possibly also a positive check: at least one entry must publish `*.dylib` files when the consuming recipe has a `homebrew` step on darwin.
- New: error when `runtime_dependencies` overlaps with step-level `dependencies` (the wget duplication anti-pattern).

### Cross-platform consistency

The field name reads identically. The semantic is "these libs are chained into the binary's resolver path":

- macOS: install_name_tool -add_rpath against bottle dylibs.
- Linux ELF: equivalent walk emitting `$ORIGIN/../lib:{libs_dir}/<dep>-<ver>/lib` (the design will need this anyway per Lead 4 in the exploration findings).

So one field, one mental model, both platforms. Good fit if the overload is acceptable.

## Option (b): Reuse per-step `dependencies`

### TOML shape

`wget.toml`:

```toml
[metadata]
name = "wget"

[[steps]]
action = "homebrew"
formula = "wget"
dependencies = ["openssl", "gettext", "libidn2", "libunistring"]
when = { os = ["darwin"] }

[[steps]]
action = "homebrew"
formula = "wget"
when = { os = ["linux"], libc = ["glibc"] }
# no dependencies needed on Linux glibc - bottle's deps are already absorbed
```

`tmux.toml` darwin:

```toml
[[steps]]
action = "homebrew"
formula = "tmux"
dependencies = ["libevent", "utf8proc"]
when = { os = ["darwin"] }
```

### Flow to ExecutionContext

Already wired correctly. `resolver.go:117-122` puts step-level `dependencies` into `ResolvedDeps.InstallTime`. `executor/plan_generator.go:704` expands them. `executor.go:964-973` populates `ctx.Dependencies.InstallTime`. `homebrew_relocate.go:609` reads them.

The only change needed is: **lift the `Type == "library"` gate** in `homebrew_relocate.go:103` and add the Linux ELF equivalent. No new wiring â€” this is the path that already works for wget today.

### Backward compatibility

Cleanest of the three. Step-level `dependencies` is rarely set on `homebrew` steps (not auto-generated), so the lifted gate would be a near no-op for current recipes. The only behavior change: any current `homebrew` step on macOS that has `dependencies = [...]` and is a non-library recipe (currently just wget) gets the chaining for free. Good â€” that's the bug fix.

### Author ergonomics

- The duplication goes away (wget shrinks to one list).
- BUT: the meaning of "step-level dependencies" gets a hidden second job. It used to mean "install these before this step runs" â€” purely install-time. Now it also means "chain these into the binary." Authors writing a `configure_make` step with `dependencies = ["openssl"]` aren't writing a load-time declaration; they want a build-time one. The chaining walk needs to be specifically tied to actions that produce binaries needing relocation (homebrew bottles, prebuilt-binary actions).
- Per-step is also wrong-grain for cross-platform recipes: tmux's `homebrew` step on darwin needs `libevent`, but the linux `homebrew` step doesn't (Linux bottles are typically self-contained). Authors must duplicate `dependencies` per-platform-step instead of declaring them once at metadata level. wget already shows this pain â€” see line 36.
- "dependencies" as a field name doesn't say "load path" or "chain." Authors won't intuit it.

### Validator implications

- Already validates that each entry is a known recipe.
- New: warn that step-level `dependencies` on a `homebrew` step means dylib chaining (on macOS) â€” surface the implicit behavior.
- Hard to express "this step's deps are runtime/chain deps" vs "this step's deps are pre-build inputs" â€” they look identical.

### Cross-platform consistency

The field reads identically per-step. But because chaining is platform-specific (matters for macOS dylib RPATHs and Linux ELF `$ORIGIN` walks but not for Alpine apk packages), authors end up writing the same `dependencies = [...]` on a darwin-gated step and a glibc-gated step (or just darwin), which is exactly the duplication wget shows today. The interface doesn't reduce author burden across platforms.

## Option (c): New explicit field (`chained_lib_dependencies`)

### TOML shape

Metadata-level:

```toml
[metadata]
name = "wget"
chained_lib_dependencies = ["openssl", "gettext", "libidn2", "libunistring"]

[[steps]]
action = "homebrew"
formula = "wget"
when = { os = ["linux"], libc = ["glibc"] }

[[steps]]
action = "homebrew"
formula = "wget"
when = { os = ["darwin"] }
```

`tmux.toml`:

```toml
[metadata]
name = "tmux"
# Linux bottle is self-contained; macOS bottle needs chained libs from tsuku
chained_lib_dependencies = ["libevent", "utf8proc"]
```

Could also exist at step level (`Step.ChainedLibDependencies`) for finer control if a recipe has multiple binaries from multiple sources, but metadata-level is sufficient for every observed case.

### Flow to ExecutionContext

New field; new wiring:

1. `recipe.MetadataSection.ChainedLibDependencies []string`.
2. `resolver.go` adds a third map to `ResolvedDeps`: `ChainedLibs map[string]string`.
3. `executor/plan_generator.go` expands `ChainedLibs` into install plans (same as `InstallTime` â€” these libs must be installed for chaining to work).
4. `executor.go` populates `ctx.Dependencies.ChainedLibs` (or merges into a unified `ctx.Dependencies.InstallTime` since the chaining walk reads from that).
5. `homebrew_relocate.go` and the new Linux ELF walker explicitly read `ctx.Dependencies.ChainedLibs` (or read `InstallTime` after the merge).

Implementation cost is the highest of the three; ~5 files touched plus tests. Conceptually clean: `Runtime` keeps its "wrapper script PATH shim" meaning, `Dependencies` keeps its "install-time inputs" meaning, `ChainedLibs` gets the new "load-time linked libs" meaning.

### Backward compatibility

100% additive. Zero existing recipes use the field. No semantic shift for `runtime_dependencies` or step `dependencies`. Library recipes continue to work via the existing path; the `Type == "library"` gate can stay, with the new walk gated on `ChainedLibDependencies` non-empty (or on an explicit set, no implicit derivation).

### Author ergonomics

- The name is self-documenting. "chained_lib_dependencies" or "chained_dylibs" tells the author exactly what the field does. Authors don't need to know about install_name_tool or RPATH semantics.
- Single declaration site; no platform duplication. Authors write it once at metadata and the engine handles macOS @rpath / Linux $ORIGIN translation.
- Cost: another field for authors to learn. Five recipe files use the wget pattern today; thirteen-ish more likely will. The field is niche.
- A new field is also a new opportunity for misuse (declaring chained_lib_dependencies on a recipe whose binary doesn't actually link those libs). Validator can catch this with `otool -L` / `readelf -d` checks during validation.

### Validator implications

- Each entry must be a known recipe.
- Each entry should have `Type == "library"` OR publish dylibs/sos in its `lib/` directory.
- Optional sandbox check: after install, run `otool -L $binary` (macOS) / `readelf -d $binary` (Linux) and verify each declared chained lib is referenced by the binary. Catches typos and over-declaration.
- Forbid declaring `chained_lib_dependencies` on recipes whose only platform path is a wrapper-style action (`npm_install`, `pipx_install`) â€” the walk doesn't apply.

### Cross-platform consistency

The field is platform-neutral by name. The engine maps it to:

- macOS: install_name_tool -add_rpath
- Linux: patchelf --set-rpath with $ORIGIN-relative entries

Reads the same on both platforms. Authors don't need to know which mechanism applies. Best of the three on this dimension.

| Dimension | (a) runtime_dependencies | (b) step dependencies | (c) chained_lib_dependencies |
|---|---|---|---|
| Wiring change scope | Medium (Runtime â†’ InstallTime merge or duplicate walk) | Tiny (lift Type gate only) | Medium (new field, new map, ~5 files) |

## Recommendation: Option (c) â€” new `chained_lib_dependencies` field

### Why

1. **Backward compatibility is decisive.** 323 recipes use `runtime_dependencies` today. Most exist solely to drive wrapper-script generation (auto-emitted by the homebrew builder). Repurposing the field to also drive a macOS install_name_tool / Linux patchelf walk is a silent behavior change across the entire registry. Even if the chaining walk is no-op for irrelevant deps, it's still a "we secretly added a load-bearing semantic to a field 323 recipes already use" â€” exactly the class of change the constraint "Schema additions OK; breaking changes are not" was written to prevent.

2. **Step-level `dependencies` is the wrong grain.** Chained libs are a property of "the binary that gets installed," not "the step that produced it." A recipe might have multiple steps (homebrew + install_binaries + verify) but only one chain. Putting the declaration at step level forces authors to either pick "the right step" arbitrarily or duplicate per platform-gated step (the wget pain that motivated this decision in the first place).

3. **Explicit > overloaded.** The wiring bug the exploration found exists precisely because `runtime_dependencies` was overloaded â€” `install/manager` reads it as map-of-versions for wrapper PATH, `homebrew_relocate` would need to read it as list-of-libs for RPATH chaining, and `executor` doesn't expand it into install plans at all. Adding a third meaning to an already-confused field deepens the confusion. A purpose-built field has a single consumer, a single producer, and a single mental model.

4. **Validator power.** With a dedicated field, the validator can perform a strong postcondition check: after sandbox install, run `otool -L` / `readelf -d` against the produced binary and verify every declared chained lib actually appears in the load commands. With `runtime_dependencies` or step `dependencies`, the validator can't tell whether a missing reference is a bug or a wrapper-PATH-only declaration.

5. **Author intuition.** Recipe authors approaching this for the first time read "chained_lib_dependencies" and immediately understand: these libs get linked into my binary's load path. They read "runtime_dependencies" and have to grep the codebase to figure out whether that means PATH, dylib chain, or both.

The exploration findings ranked Option 2 (strengthen the homebrew action automatically) as the leading shape because it minimized recipe-author burden. (c) is compatible with that: the homebrew action still auto-runs the chaining walk, but reads from an unambiguous source. The author burden of "declare one new field at metadata level" is lower than the burden of "remember which of two existing fields you needed and whether to duplicate it across platform-gated steps" (today's wget reality).

### Implementation notes

- Add `MetadataSection.ChainedLibDependencies []string` and matching `ExtraChainedLibDependencies` for extension. No step-level variant initially â€” observed cases all fit metadata level.
- `ResolvedDeps` gains a third map (or merge into `InstallTime` at resolver time and tag the source). Either works; the merge approach minimizes call-site changes downstream.
- The chaining walk is gated on `ChainedLibDependencies` non-empty rather than on `Type == "library"`. Library recipes that already publish dylibs continue to use the existing implicit path (no field needed); tool recipes opt in explicitly.
- Validator: warn when an entry's recipe doesn't publish dylibs or sos. Strong postcondition check via `otool -L` / `readelf -d` runs in sandbox validation.
- Linux equivalent of `fixLibraryDylibRpaths` is a separate decision (Decision 3 territory) but the field's interface is platform-neutral, so it's ready when that walker lands.

### What we give up

- One additional field for authors to learn. Mitigation: skill update in `tsuku-recipe-author` (per the Plugin Maintenance section in CLAUDE.md), with examples on the wget/tmux migration.
- Slightly more wiring than (b). Worth it for the explicit semantics and validator power.
