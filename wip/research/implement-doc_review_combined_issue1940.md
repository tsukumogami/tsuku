# Combined Review: Issue #1940 -- RubyGems gem-based executable discovery

## Summary

The implementation correctly mirrors the PyPI wheel-based pattern from #1939. Artifact-based discovery from `.gem` metadata.gz is sound, the fallback chain is well-tested, and `AuthoritativeBinaryNames()` properly returns nil for non-artifact paths. Three advisory findings, one blocking.

## Blocking Findings

### 1. Dead `serverURL` variable pattern in tests (5 occurrences)

`/home/dangazineu/dev/workspace/tsuku/tsuku-2/public/tsuku/internal/builders/gem_artifact_test.go` lines 73-95, 177-199, 244-266, 300-322, 623-645.

```go
var serverURL string
// ...
serverURL = server.URL
_ = serverURL
```

This 3-line pattern appears in 5 test functions. The variable is declared, assigned, and immediately silenced. It's dead code. In TLS server tests `newGemBuilderWithTLSServer(server)` extracts the URL internally, so `serverURL` is never used. **Remove all 5 occurrences.**

**Severity: Blocking** -- dead code committed to test infrastructure that will be cargo-culted into future builder tests.

## Advisory Findings

### 2. `extractTarEntry` uses `maxGemDownloadSize` as read limit instead of entry-specific limit

`/home/dangazineu/dev/workspace/tsuku/tsuku-2/public/tsuku/internal/builders/gem.go:323`

```go
content, err := io.ReadAll(io.LimitReader(tr, maxGemDownloadSize))
```

The tar entry reader uses the 50MB gem download limit, but the only caller extracts `metadata.gz` which is then decompressed with a 1MB limit. The 50MB limit on the compressed entry is generous but not wrong -- the gzip decompression limit catches oversized content. Noting for awareness only.

**Severity: Advisory** -- correct but the limit could be `maxGemMetadataSize` for tighter bounds. Not blocking since the decompression limit catches it.

### 3. Line-based YAML parser does not handle quoted strings

`/home/dangazineu/dev/workspace/tsuku/tsuku-2/public/tsuku/internal/builders/gem.go:359-409`

The `parseGemMetadataExecutables` parser handles the common YAML patterns (list items, inline values, empty arrays). However, it doesn't handle quoted executable names in list items (e.g., `- "bundle"`). In practice, RubyGems metadata.gz always uses unquoted names since the YAML is machine-generated by `gem build`. The `isValidExecutableName` filter would also reject names with quotes. This is safe for production data.

**Severity: Advisory** -- real-world gem metadata doesn't quote executable names.

## Pattern Compliance

The implementation correctly follows the #1939 PyPI pattern:
- Same `discoverExecutables` -> artifact -> fallback chain structure
- Same `cachedXxxExecutables` + `AuthoritativeBinaryNames()` pattern
- Same `downloadArtifact` shared helper usage
- Same `fromArtifact` bool to gate caching
- Interface compliance test (`var _ BinaryNameProvider = (*GemBuilder)(nil)`)

## Structural Notes

- `extractTarEntry` and `decompressGzip` are package-level helpers with their own unit tests. These are general-purpose and could serve future builders (e.g., Homebrew bottles are tar.gz). Reasonable to extract rather than inline.
- The gem metadata YAML parser avoids a `gopkg.in/yaml.v3` dependency for a single use case. The tradeoff is sound -- gem metadata YAML is machine-generated with a stable format.
