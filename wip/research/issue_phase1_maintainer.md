# Maintainer Assessment: update-registry freshness bypass bug

## Issue description quality

**Clearly defined?** Mostly yes, with one gap.

The symptom is specific and reproducible: after a PR merges a recipe change, `tsuku
update-registry` reports "infisical: already fresh" and the stale cached recipe
remains in use. The reproduction path is short (PR merges, user runs
`update-registry`, installs tool, sees old behaviour). The gap is that the report
does not state how recently infisical was previously installed or cached -- that
detail matters for understanding why the TTL has not expired yet, but it is not
required to diagnose the root cause.

**Type:** bug

**Scope:** yes -- single, well-bounded defect in the TTL-based freshness check.

---

## Root cause (from code analysis)

### How `update-registry` decides to skip a recipe

`runRegistryRefreshAll` (cmd/tsuku/update_registry.go:147) calls
`cachedReg.RefreshAll(ctx)` when `--all` is not passed.

`RefreshAll` (internal/registry/cached_registry.go:316) iterates cached recipes and
calls `c.isFresh(meta)` for each one. When that returns true, it appends a
`RefreshDetail{Status: "already fresh"}` and skips the network fetch entirely.

`isFresh` (internal/registry/cached_registry.go:195):

```go
func (c *CachedRegistry) isFresh(meta *CacheMetadata) bool {
    expiresAt := meta.CachedAt.Add(c.ttl)
    return time.Now().Before(expiresAt)
}
```

The freshness check is **time-only**. It has no awareness of remote content. If the
recipe was cached less than 24 hours ago (the default TTL set in
`config.DefaultRecipeCacheTTL`), it is treated as fresh regardless of what changed
upstream.

### Why a merged PR does not invalidate the cache

The registry serves raw files from
`https://raw.githubusercontent.com/tsukumogami/tsuku/main/recipes/...`. When a PR
merges, the file on `main` changes immediately. Nothing pushes an invalidation signal
to user machines. There is no ETag, no `Last-Modified` check, no manifest hash
comparison -- the HTTP client in `fetchRemoteRecipe` issues a plain GET with no
conditional headers.

The manifest (`recipes.json`) is fetched during `update-registry` (via
`refreshManifest`), but `ManifestRecipe` only carries name, description, homepage,
dependencies, and satisfies fields. It does not carry a per-recipe content hash or
version. The manifest therefore cannot be used to detect that a recipe's content
changed.

### Why `--all` is not a workaround users know to reach for

The flag exists (`--all` forces refresh of all cached recipes regardless of
freshness) but is not surfaced in the default output or help text in a way that
guides a user who just ran `update-registry` and saw "already fresh". A user who
sees that message has no indication that their cache could be stale relative to the
remote.

---

## The specific scenario that triggers this bug

1. User installs infisical for the first time. Recipe is cached and `CachedAt` is
   written to `i/infisical.meta.json`.
2. A PR merges fixing the infisical recipe.
3. User runs `tsuku update-registry` within 24 hours of step 1.
4. `isFresh` returns true. `RefreshAll` skips infisical. User sees "already fresh".
5. `tsuku install infisical` reads the now-stale cached TOML and hits the same bug.

---

## Fix options (for issue spec, not implementation decision)

Three approaches exist, from narrowest to broadest:

**A. Content-hash comparison in RefreshAll**
Store the SHA-256 of each cached recipe (already available in `CacheMetadata.ContentHash`).
During `update-registry`, fetch fresh content for all cached recipes regardless of TTL,
compare hashes, and update the cache only when the hash differs. This is accurate but
requires one HTTP GET per cached recipe on every `update-registry` run.

**B. Add content hash to the manifest**
Extend `ManifestRecipe` with a `content_hash` field generated by
`scripts/generate-registry.py`. During `update-registry`, compare the manifest hash
for each locally cached recipe. Recipes whose hash differs are refreshed. This costs
one manifest fetch instead of N recipe fetches, but requires regenerating the manifest
on every recipe change (which CI already does).

**C. Expose `--all` more clearly / document workaround**
Document that `update-registry --all` forces a full refresh. This is the lowest effort
but does not fix the underlying issue -- users still must know to use the flag.

Option B is the most efficient and fits the existing manifest infrastructure.
Option A is the most defensive and works even if the manifest is unavailable.
Both are worth specifying; option C is a partial mitigation only.

## Gaps and clarifications needed

1. **PR number and recipe name are given, but the actual bug in the recipe is not
   described.** The issue should include what the pre-PR recipe did wrong so a
   reviewer can verify the fix works end-to-end.
2. **User's cache age is not stated.** Knowing whether the user ran `update-registry`
   within 24 h of first installing infisical (or a long time ago with a fresh install
   in between) would confirm this is a TTL bypass rather than a different code path.
3. **Whether `--all` was tried** is not mentioned. Noting that outcome would help
   distinguish a TTL bypass from a deeper caching defect.

None of these gaps block filing the issue -- the root cause is clear from the code.
