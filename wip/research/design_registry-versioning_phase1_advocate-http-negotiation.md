# Advocate: HTTP Version Negotiation

## Approach Description

The CLI and registry negotiate schema compatibility via HTTP-level mechanisms. Concretely:

1. **CLI sends an `Accept` header** indicating the schema versions it supports, e.g., `Accept: application/vnd.tsuku.manifest.v1+json, application/vnd.tsuku.manifest.v2+json`.
2. **Registry responds with a `Content-Type` header** declaring the schema version of the payload, e.g., `Content-Type: application/vnd.tsuku.manifest.v2+json`.
3. **Smart registries** (HTTP APIs) can serve different payload formats based on the `Accept` header. During migration, they serve v1 to old CLIs and v2 to new CLIs.
4. **Static registries** (GitHub raw, Cloudflare Pages) can't do content negotiation, so they fall back to the in-manifest `schema_version` field. The CLI checks the response `Content-Type` first; if it's generic `application/json`, it falls through to manifest-level validation.
5. **Versioned URL paths** as an alternative: `https://tsuku.dev/v1/recipes.json` vs `https://tsuku.dev/v2/recipes.json`. The CLI tries its preferred version first, falls back to the lower version.

The deprecation lifecycle works as follows:
- Phase 1: Registry starts including a `deprecation` object in the v1 manifest (additive, safe for old CLIs).
- Phase 2: New CLIs read the deprecation metadata and warn users.
- Phase 3: Smart registries start serving v2 to CLIs that accept it, v1 to those that don't.
- Phase 4: Registry stops serving v1. Old CLIs get 406 Not Acceptable (smart) or parse failures caught by manifest-level version check (static).

## Investigation

### Current HTTP client usage

The registry's HTTP client is created in `newRegistryHTTPClient()` (`internal/registry/registry.go:37-51`). It's a standard `http.Client` with security-conscious transport settings (disabled compression, proper timeouts). No custom headers are set on requests.

Manifest fetching (`internal/registry/manifest.go:77-137`) creates a bare `http.NewRequestWithContext` with no `Accept` header. The response is consumed as raw bytes with no `Content-Type` inspection. The `parseManifest` function (line 158-164) does a plain `json.Unmarshal` with zero validation of `SchemaVersion`.

Recipe fetching (`internal/registry/registry.go:140-198`) follows the same pattern: no custom headers, no content-type inspection.

### The static hosting problem

The central registry manifest is served from `https://tsuku.dev/recipes.json` (line 20 of manifest.go). This is generated by `scripts/generate-registry.py` and deployed to Cloudflare Pages as a static file. Cloudflare Pages has no server-side logic for content negotiation -- it serves whatever file is at the path.

This is the core tension of this approach. The central registry today is a static file. Content negotiation requires either:
- A Cloudflare Worker sitting in front of the static file to inspect `Accept` headers and route to different files
- Versioned URL paths (`/v1/recipes.json`, `/v2/recipes.json`) where the generation script produces multiple files

### Discovery registry contrast

The discovery registry (`internal/discover/registry.go:47-59`) uses integer `SchemaVersion` with a hard check: `if reg.SchemaVersion != 1`. This is the proven in-codebase pattern. It requires no HTTP infrastructure -- the version is in the payload itself. HTTP negotiation would be a second, parallel mechanism.

### Distributed registries

Future distributed registries (#2073) may be git repos, static file servers, or HTTP APIs. Git-based registries (like Homebrew taps) have no HTTP negotiation capability at all. Static file servers (GitHub Pages, S3) have limited or no server-side logic. Only full HTTP API registries could implement proper content negotiation.

### Cache interaction

The manifest cache (`$TSUKU_HOME/registry/manifest.json`) is written as raw bytes (`CacheManifest`, line 140-155). No HTTP headers are cached alongside it. If the CLI relies on HTTP headers for version information, that signal is lost when reading from cache. The cached manifest would need a sidecar or wrapper to preserve the negotiated version.

## Strengths

- **Elegant for smart registries.** An HTTP API registry can serve multiple schema versions simultaneously with zero downtime. Old CLIs get old format, new CLIs get new format. The migration window is entirely server-controlled. This is the gold standard for API versioning.

- **Standard protocol.** Content negotiation via `Accept`/`Content-Type` is well-understood HTTP semantics (RFC 7231 Section 5.3). Developers running custom registries backed by HTTP APIs would find this natural.

- **Clean separation of concerns.** The transport layer handles compatibility negotiation; the parsing layer handles data. The CLI's `parseManifest` function doesn't need to handle multiple format versions -- it only ever sees the format it requested.

- **Graceful degradation signal.** A 406 Not Acceptable response is an unambiguous machine-readable signal that the CLI doesn't speak any version the registry supports. No parsing needed to detect incompatibility.

- **Versioned URL paths work without server logic.** The URL-path variant (`/v1/recipes.json`, `/v2/recipes.json`) works with static hosting. The generation script produces both files during the migration window. The CLI tries its preferred version URL, and a 404 means "this version isn't available."

## Weaknesses

- **Central registry is static.** The `Accept`-header approach requires server-side logic that `tsuku.dev` (Cloudflare Pages) doesn't have. Either a Cloudflare Worker must be added, or the approach must fall back to versioned URL paths for static registries. This means the "elegant" path is unavailable for the primary use case today.

- **Cache loses the signal.** `CacheManifest` (line 140-155) stores raw JSON bytes with no HTTP metadata. Reading from cache bypasses HTTP entirely, so the CLI must fall back to manifest-level `schema_version` anyway. Every code path that reads cached data needs the fallback, which means the HTTP mechanism is supplementary rather than primary.

- **Two mechanisms, not one.** Since static/git registries can't do HTTP negotiation, the CLI must implement manifest-level version checking as a fallback regardless. The HTTP negotiation becomes an optimization on top of the mandatory base mechanism, increasing surface area for bugs.

- **Distributed registries are mostly static.** Git repos served via GitHub raw URLs, S3 buckets, or similar static hosting are the likely shape of distributed registries. These can't inspect `Accept` headers. The approach that only works for HTTP APIs serves the minority of registries.

- **Versioned URL paths have their own costs.** If using `/v1/` and `/v2/` paths instead of header negotiation: the generation script must produce multiple manifest versions; the CLI must handle 404 fallback logic; cache paths must be version-aware; and the `TSUKU_MANIFEST_URL` env var override becomes ambiguous (does the user specify `/v1/` or the base?).

- **Breaking change to add it.** Adding `Accept` headers to requests is not breaking, but changing the URL structure (versioned paths) or adding a Worker layer is infrastructure complexity. The `DefaultManifestURL` constant (`manifest.go:20`) and `TSUKU_MANIFEST_URL` override would need redesign.

## Deal-Breaker Risks

- **The fallback IS the feature.** Since every non-HTTP code path (cache reads via `GetCachedManifest`, local registries via `isLocal` path in `FetchManifest`, git-based distributed registries) must use manifest-level version checking, the HTTP negotiation layer provides marginal value on top of the thing you have to build anyway. The risk is building and maintaining two version-checking mechanisms when one suffices.

- **Additive deployment constraint.** Old CLIs don't send `Accept` headers. If the registry moves to versioned URL paths, old CLIs hitting the old URL must still work. This requires maintaining the unversioned URL indefinitely or accepting that the mechanism itself causes breakage for old CLIs -- directly violating the "additive deployment" decision driver.

## Implementation Complexity

### Files to modify

- `internal/registry/manifest.go` -- Add `Accept` header to manifest fetch request; inspect `Content-Type` on response; implement versioned URL path logic or header-based version detection
- `internal/registry/registry.go` -- Add `Accept` header to recipe fetch requests; version-aware URL construction
- `internal/registry/manifest.go` (`CacheManifest` / `GetCachedManifest`) -- Store version metadata alongside cached manifest (sidecar file or wrapper JSON)
- `scripts/generate-registry.py` -- Generate manifest at multiple versioned paths if using URL-path strategy
- Website/Cloudflare deployment -- Add Worker for content negotiation, or restructure static output to include versioned directories

### New infrastructure

- Cloudflare Worker (if using `Accept` header negotiation) OR versioned directory structure in static site output
- Cache sidecar or wrapper to preserve negotiated version information
- Fallback mechanism to manifest-level `schema_version` for all non-HTTP paths

### Estimated scope

Medium-large. The HTTP header changes to the Go client are straightforward (maybe 50 lines across manifest.go and registry.go). But the infrastructure changes (Worker or versioned URLs), cache version-awareness, and the mandatory manifest-level fallback add significant scope. The fallback alone is roughly equivalent to the simpler "manifest-level version check" approach, making the HTTP layer purely incremental complexity.

## Summary

HTTP version negotiation is the technically correct approach for API-backed registries and provides the cleanest migration experience when the server can serve multiple formats simultaneously. However, tsuku's central registry is a static file on Cloudflare Pages, and distributed registries will likely be static or git-based too. Since every non-HTTP code path (cache, local, git) requires manifest-level version checking as a fallback, the HTTP negotiation layer becomes supplementary complexity on top of the mechanism you must build anyway. This approach makes the most sense if tsuku's registry is expected to move to a dynamic HTTP API in the near future; for the current static-file architecture, it adds infrastructure cost with limited incremental benefit.
