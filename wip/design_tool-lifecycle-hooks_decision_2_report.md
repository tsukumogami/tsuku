### Decision: How per-tool shell init scripts reach the user's shell

**Context**

Tsuku's `shellenv` command outputs a static `export PATH=...` line. Some tools
need more than PATH -- direnv, zoxide, niwa, and starship all require eval-init
wrappers that define shell functions, hook into `cd`, or customize prompts.
Exploration already decided on a shell.d directory model: post-install hooks
write init scripts to `$TSUKU_HOME/share/shell.d/{tool}.{shell}`, and these
are sourced at shell startup.

The open question is the delivery mechanism: how do those shell.d files get
sourced, how is startup time kept under 5ms, and what happens to stale scripts
when tools are removed?

Three constraints shape this decision. First, the activation hook already runs
`tsuku hook-env` on every prompt via PROMPT_COMMAND/precmd/fish_prompt -- adding
per-tool init to the prompt path is unacceptable (each tool's init costs
10-30ms). Second, tsuku already manages hook files in `$TSUKU_HOME/share/hooks/`
using an embedded-file-write + rc-marker-block pattern that's proven and
idempotent. Third, tools must work standalone -- a user who installs niwa
outside tsuku should still be able to run `eval "$(niwa init bash)"` directly.

**Assumptions**

- Shell.d scripts are small (under 50 lines each). If a tool's init is
  expensive, the tool itself is responsible for lazy-loading. Tsuku's caching
  covers concatenation cost, not individual tool init cost.
- The number of tools needing shell init stays in the 5-15 range. If it grows
  past 30+, the concatenated cache file approach still works but startup time
  from executing the cached content may need profiling.
- Users already have `eval "$(tsuku shellenv)"` in their rc file. Changing
  shellenv's output is a backward-compatible extension, not a breaking change,
  because the new output is additive (PATH export + source line).

**Chosen: Option A -- Extend shellenv to source shell.d/**

`tsuku shellenv` gains a second output section. After the existing PATH export,
it emits a source command for a cached combined init file:

```bash
# Current output (preserved):
export PATH="$TSUKU_HOME/bin:$TSUKU_HOME/tools/current:$PATH"

# New output (appended):
. "$TSUKU_HOME/share/shell.d/.init-cache.bash"
```

The cache file (`$TSUKU_HOME/share/shell.d/.init-cache.{bash,zsh,fish}`)
concatenates all per-tool scripts in the shell.d directory. It's regenerated by
tsuku whenever the tool list changes (install, remove, update). If the cache
file doesn't exist (no tools need shell init), the source line is omitted.

Directory structure:

```
$TSUKU_HOME/share/shell.d/
  direnv.bash
  direnv.zsh
  direnv.fish
  niwa.bash
  niwa.zsh
  zoxide.bash
  zoxide.zsh
  zoxide.fish
  starship.bash
  starship.zsh
  starship.fish
  .init-cache.bash    # concatenation of all *.bash files
  .init-cache.zsh     # concatenation of all *.zsh files
  .init-cache.fish    # concatenation of all *.fish files
```

Cache rebuild trigger: `tsuku install`, `tsuku remove`, and `tsuku update`
call a `RebuildShellCache(shell)` function after any tool with shell.d scripts
is modified. The rebuild reads all `*.{shell}` files in shell.d, sorts them
(or uses a configured order from `$TSUKU_HOME/config.toml`), concatenates
them, and atomically writes the cache file using the existing `atomicWrite`
helper from `internal/hook/install.go`.

Multi-shell support:

- **bash/zsh**: shellenv outputs `. "$TSUKU_HOME/share/shell.d/.init-cache.bash"`
  (or `.zsh`). Shell detection uses the same logic as `tsuku hook install`
  today -- the shell argument is passed or inferred from `$SHELL`.
- **fish**: shellenv outputs `source "$TSUKU_HOME/share/shell.d/.init-cache.fish"`.
  Fish syntax differs (no `.` builtin, `set -gx` instead of `export`), so
  per-tool scripts and the cache file use fish-native syntax.

Tool removal cleanup: when `tsuku remove` deletes a tool, it also deletes
`$TSUKU_HOME/share/shell.d/{tool}.{bash,zsh,fish}` and triggers a cache
rebuild. Because cleanup paths are stored in state at install time (per
exploration decision), the remove flow knows which shell.d files to delete
without consulting the recipe registry.

Startup time analysis:

| Phase | Cost |
|-------|------|
| `eval "$(tsuku shellenv)"` | ~3ms (Go binary startup + print) |
| `. .init-cache.bash` | <1ms (file read, no subprocess) |
| Executing cached content | 1-5ms (shell functions, no subshells) |
| **Total** | **~4-8ms** |

The 5ms budget applies to the incremental cost of shell.d sourcing, not the
full shellenv eval. The cache file is pure shell code (function definitions,
variable exports) -- no subprocess calls. Each tool's recipe is responsible
for generating init scripts that avoid subprocess calls at source time
(e.g., `niwa init --fast bash` could emit pre-computed shell functions rather
than calling niwa at runtime).

When the cache is stale (tool installed but cache not yet rebuilt), the source
line either hits an empty/missing file (no-op) or sources the previous cache
(missing only the new tool). The next shell session after rebuild picks up
the change. This is acceptable because shell init takes effect on new shells,
not running ones.

**Rationale**

Option A wins on three fronts.

First, it preserves the single-line user setup. Users already have
`eval "$(tsuku shellenv)"` in their rc files. No additional lines are needed.
Options B and C both require users to add something new -- B needs a second
eval line, C needs a glob-source loop. Since tsuku's install script already
configures shellenv, extending it means zero additional user friction for both
new and existing users.

Second, it reuses proven infrastructure. The hook installation system already
writes managed files to `$TSUKU_HOME/share/hooks/` and sources them via
marker blocks. Shell.d follows the same pattern: tsuku writes files, tsuku
sources them, tsuku cleans them up. The `atomicWrite` helper, the
idempotent marker approach, and the per-shell file naming convention all
transfer directly.

Third, the cached combined script approach keeps startup time predictable.
Unlike Option C (direct sourcing), which requires a glob loop on every shell
startup, Option A's cache means a single file read. Unlike Option B, which
adds a second Go binary invocation (~3ms), Option A embeds the source line
in shellenv's existing output -- no additional process spawn.

The main trade-off is coupling: shellenv now does more than PATH. But shellenv
is already the "configure your shell for tsuku" command. Adding tool init
sourcing is a natural extension of that contract, not a violation.

**Alternatives Considered**

- **Option B: New `tsuku shell-init` command**. Separate command that outputs
  concatenated tool init scripts. Users add `eval "$(tsuku shell-init)"` as a
  second line in their rc file. Same caching approach as Option A. Rejected
  because it doubles user setup friction (two eval lines instead of one),
  adds a second Go process spawn on every shell startup (~3ms), and splits
  a conceptually unified operation ("set up my shell for tsuku") into two
  commands. The separation buys clarity for developers reading the code, but
  that doesn't justify the user-facing cost. If shellenv grows too complex
  internally, it can delegate to a `shell-init` subpackage without exposing
  a separate command.

- **Option C: Direct sourcing without tsuku mediation**. Post-install hooks
  write scripts to shell.d. Users add
  `for f in $TSUKU_HOME/share/shell.d/*.bash; do source "$f"; done` to their
  rc file. No tsuku command at startup, no cache needed. Rejected for three
  reasons. (1) User setup friction is highest -- the loop syntax varies per
  shell and is error-prone (fish uses `for f in .../*.fish; source $f; end`).
  (2) No ordering control -- glob iteration is alphabetical, which may not
  match required init order (direnv before niwa, starship last). (3) Startup
  cost grows linearly with tool count because each file is a separate `source`
  call with filesystem overhead, whereas the cached approach is a single read.
  The approach does have a genuine advantage: zero dependency on tsuku at
  shell startup. If tsuku's binary is broken or missing, shells still work.
  But this edge case doesn't outweigh the daily friction.

**Consequences**

What becomes easier:
- Installing tools that need shell integration is seamless. `tsuku install direnv`
  writes shell.d scripts and rebuilds the cache. The user's next shell session
  has direnv active with no manual steps.
- Removing tools is clean. `tsuku remove direnv` deletes shell.d files, rebuilds
  cache, and the next shell session no longer sources direnv init.
- Existing users get the feature automatically -- their `eval "$(tsuku shellenv)"`
  line starts sourcing shell.d content after upgrading tsuku.

What becomes harder:
- shellenv's output is no longer a single static line. Testing and debugging
  require checking both the PATH export and the cache file content.
- Cache staleness is a new failure mode. If `RebuildShellCache` fails silently,
  users may not get tool init until the next install/remove triggers a rebuild.
  Mitigation: `tsuku doctor` should verify cache freshness.
- Tools that need per-prompt behavior (like starship's prompt customization)
  can't use shell.d for that part -- they still need integration with the
  activation hook or their own PROMPT_COMMAND registration. Shell.d handles
  one-time init only.
