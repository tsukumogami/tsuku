# Explore Scope: install-claude

## Visibility

Public

## Issue

#2252: install: "claude" resolves to unrelated ecosystem packages instead of Claude Code

## Core Question

Which major developer tools lack high-quality handcrafted recipes in the tsuku registry, what criteria distinguish a tool that warrants one, and where should these curated recipes live relative to auto-generated pipeline output? The claude disambiguation bug is the concrete first case — its solution should be consistent with the broader system design.

## Context

The batch pipeline covers a wide surface of tools via ecosystem probing and LLM-assisted generation, but focuses on breadth. Most users only install a handful of very popular tools (language runtimes, cloud CLIs, editors). These tools deserve handcrafted, periodically tested recipes. The question of where they live (embedded registry, regular recipes/, or a distinct curated tier) is open. Claude Code's npm package (`@anthropic-ai/claude-code`) is the motivating example: no recipe, no discovery entry, five unrelated package matches when users type `tsuku install claude`.

## In Scope

- Gap analysis of top ~50 developer tools vs current recipe coverage
- Criteria for what makes a tool "major" (deserving a handcrafted recipe)
- Where curated recipes live (embedded, regular, tiered)
- Periodic testing model for curated recipes
- How the claude-specific problem resolves as the first concrete case
- Comparable approaches in other package managers

## Out of Scope

- Improving the auto-generation pipeline for long-tail tools
- Changing disambiguation UI
- Changing ecosystem probe ranking

## Research Leads

1. **What does current recipe coverage look like for the top ~50 developer tools by adoption?**
   Gap analysis across the most widely-used CLIs, runtimes, and infra tools. Which are covered, missing, or have thin autogenerated recipes?

2. **What criteria distinguish a tool that warrants a handcrafted recipe?**
   Download volume, user intent (typed by name), installation complexity, platform coverage requirements. What threshold separates "curated" from "long-tail"?

3. **Where should handcrafted recipes live — embedded registry, regular recipes/, or a distinct curated tier?**
   Each option has different tradeoffs: embedded = always available but bloats binary; regular recipes/ = same pipeline, different provenance signal; curated tier = explicit contract but new infrastructure.

4. **How should periodic testing of curated recipes work?**
   The existing sandbox pipeline tests changed recipes on PR. Do curated recipes need a more frequent, independent test signal? What does "periodically tested" mean in practice?

5. **How does the claude problem specifically resolve under the broader design?**
   The recipe itself and any discovery/alias entries needed. Should be consistent with whatever curated recipe pattern emerges.

6. **What do other package managers do for popular tool curation?**
   Homebrew Core, nixpkgs maintained packages, asdf plugins — what makes their curation trustworthy, discoverable, and maintainable?
